ACP Logging API

This API was created to provide a consistent means for writing to log files across projects. It meets the following requirements:

  • Hierarchical log levels (DEV, DEBUG, INFO, WARNING, ERROR
  • Automatically capture date, time, thread ID, and function name
  • Type-safe (uses mostly string streams, as few macros as possible)
  • Thread-safe

The log levels should be used as follows:

0. DEV - For software development only. Temporary debugging statements that can be as verbose as desired.

  1. DEBUG - For debugging purposes. Used to diagnose unknown issues, permanent but not visible in production.
  2. INFO - For general information. The lowest level visible in production. For general purpose messages.
  3. WARN - For possible problems. Indicates a situation which is unexpected and could cause undesired behavior.
  4. ERROR - For definite problems. Indicates a situation which is definitely problematic. May be followed by system crash.

Usage: 

 ACP_LOG_MESSAGE(ACP_LOG_DEV) << "This is where it always breaks";
 ACP_LOG_MESSAGE(ACP_LOG_DEBUG) << "Number of signals is " << num_signals;
 ACP_LOG_MESSAGE(ACP_LOG_INFO) << "Starting GPS thread.";
 ACP_LOG_MESSAGE(ACP_LOG_WARN) << "Config file is missing expected signal ID " << signal_id << ": continuing anyway."
 ACP_LOG_MESSAGE(ACP_LOG_ERROR) << "Config file is missing!!! Shutting down...";

If you want to create multiple logging objects within your program, you can do this as well. Simply create a LogStream object, and a LogStreamLock when you wish to write:

// Example LogStream acp_daq_logger("acp_daq_logger.log", ACP_LOG_INFO); LogStream acp_hub_logger("acp_hub_logger.log", ACP_LOG_WARN);

#define ACP_DAQ_LOG(log_level, tag) LogStream::LogStreamLock(acp_daq_logger, (log_level), BOOST_CURRENT_FUNCTION, (tag)) #define ACP_HUB_LOG(log_level, tag) LogStream::LogStreamLock(acp_hub_logger, (log_level), BOOST_CURRENT_FUNCTION, (tag))

ACP_DAQ_LOG(ACP_LOG_WARN, "MY_FIRST_TAG") << "This message goes to daq log"; ACP_HUB_LOG(ACP_LOG_TEST, "MY_OTHER_TAG") << "This message goes to hub log";

A Singleton is an object which should only have one instance in a program.

This is a Singleton template implemented as a CRTP class.
CRTP (Curiously Recurring Template Pattern) means that the derived
class inherits from the base template, where the template argument
is the derived class itself.

In other words, to implement a Singleton named MySingleClass, you would
declare your class as such:

    class MySingleClass : public Singleton<MySingleClass>;

A good explanation of CRTP can be found here:
https://en.wikipedia.org/wiki/Curiously_recurring_template_pattern
